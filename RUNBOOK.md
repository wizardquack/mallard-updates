# Mallard updater feed — operations runbook

## What this repo serves

GitHub Pages serves the contents of `main` branch root at
https://mallard-updates.vnsf.xyz/. Two files are load-bearing:

- `latest.json` — Tauri v2 updater manifest, stable channel
- `latest-beta.json` — same shape, beta channel (published but not yet
  consumed by any shipping Mallard binary; see README)

The manifest carries the new version's metadata + per-platform
`{ signature, url }`. `url` points at a release asset on
`wizardquack/mallard`'s GitHub Releases page; `signature` is an inline
base64-encoded `.sig` (ed25519, generated by `tauri-action` in CI
against the `TAURI_SIGNING_PRIVATE_KEY` secret).

Installed Mallard binaries embed the corresponding **public** key at
compile time (in `src-tauri/tauri.conf.json > plugins.updater.pubkey`).
The `@tauri-apps/plugin-updater` plugin downloads the manifest, fetches
the platform-matching artifact, verifies the inline signature against
the embedded pubkey, and replaces the running binary.

## Routine release publishing

Push a tag `vX.Y.Z` (or `vX.Y.Z-rcN`) to `wizardquack/mallard`. The
`release.yml` workflow builds + signs the artifacts and the
`publish-feed` job pushes a commit here. No human action needed.

If the auto-push fails (deploy key revoked, network blip, etc.), see
"Manual republish" below.

## Key rotation

Rotate when: suspected leak; periodic best-practice (every 12-18 months).

The Tauri updater plugin supports an **array** of acceptable pubkeys.
Rotation uses that to transition without bricking existing installs:

1. Generate a new keypair locally:
   ```
   npx @tauri-apps/cli@latest signer generate -w ~/.tauri/mallard-NEW.key
   ```
2. Check whether the installed `@tauri-apps/plugin-updater` version
   supports multiple `pubkey` values (an array or newline-separated
   list). Tauri v2.10.x as of #19b ship time treats `pubkey` as a
   single string. If multi-pubkey support has landed by your rotation
   date, edit `wizardquack/mallard`'s `src-tauri/tauri.conf.json` so
   `plugins.updater.pubkey` lists the OLD pubkey followed by the NEW
   pubkey, then proceed with the transitional release. If not, the
   only path forward is a hard swap (ship a release signed by NEW,
   replace `pubkey` with NEW only) — existing installs on the old
   pubkey will be unable to auto-update and must reinstall manually.
   Plan accordingly.
3. Ship a transitional release. Existing installs verify with the OLD
   key (the new release is still signed with OLD); after they update,
   they're running a binary that trusts both.
4. After one full release cycle (long enough that ~all active users
   have updated through the transitional release), drop the OLD key
   from `pubkey`. Update the GitHub Actions secret
   `TAURI_SIGNING_PRIVATE_KEY` to the NEW private key. Ship the next
   release signed with NEW only.

## Channel maintenance

`latest.json` is updated only on non-rc tags. `latest-beta.json` is
updated on every tag (rcs + stables). The in-app updater currently
only reads `latest.json` (beta channel deferred to v1.x per #19b
reshape); `latest-beta.json` is maintained so a follow-up plan can
wire the toggle without re-bootstrap.

To promote an rc to stable without re-running the full release matrix:
re-tag the rc commit with the stable `vX.Y.Z` and push. CI re-runs and
overwrites `latest.json`.

## Manual republish

If the auto-push failed, rerun the publish-feed job:

```
gh workflow run release.yml --ref <tag>
```

Or run the script locally:

```
git clone git@github.com:wizardquack/mallard-updates.git
cd <mallard checkout>
gh auth status   # ensure logged in
node scripts/build-updater-manifest.mjs --tag <vX.Y.Z[-rcN]> --out ../mallard-updates/
cd ../mallard-updates
git add latest.json latest-beta.json
git commit -m "Republish <tag>"
git push origin main
```

## Emergency rollback (pin to a previous stable)

If a stable release ships broken:

1. `git log` this repo to find the previous-stable commit.
2. `git show <prev-commit>:latest.json > latest.json` (overwrite with
   the previous-stable manifest).
3. `git commit -am "Rollback to <prev-version>"` and push.
4. Existing installs on the broken version will downgrade on next
   check (Tauri accepts downgrades from the feed).
5. Investigate, fix, ship `vX.Y.Z+1`; the next `publish-feed` run
   overwrites the rollback with the fixed version.

## Disaster recovery

If this repo is wiped or compromised:

- **Wiped:** rebuild from a local clone (any maintainer who has cloned
  recently has the full history). Re-enable Pages + CNAME via repo
  Settings.
- **Deploy key compromise:** revoke + regenerate `MALLARD_UPDATES_DEPLOY_KEY`
  in `wizardquack/mallard`'s Actions secrets. The deploy key only
  grants push to this repo — it does NOT allow signing fake updates.
- **Tauri signing key compromise:** trigger Key Rotation above
  immediately. Existing installs trust the compromised key; the only
  defense is shipping a transitional release before an attacker can.

## Schema reference

A v2 manifest:

```json
{
  "version": "v0.2.0",
  "notes": "See https://github.com/wizardquack/mallard/releases/tag/v0.2.0",
  "pub_date": "2026-06-15T12:34:56Z",
  "platforms": {
    "darwin-aarch64": {
      "signature": "<base64 ed25519 sig>",
      "url": "https://github.com/wizardquack/mallard/releases/download/v0.2.0/Mallard_0.2.0_aarch64.app.tar.gz"
    },
    "linux-x86_64":   { "signature": "...", "url": "..." },
    "windows-x86_64": { "signature": "...", "url": "..." }
  }
}
```
